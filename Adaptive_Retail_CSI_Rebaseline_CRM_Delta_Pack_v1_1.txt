Adaptive Retail CSI Camera Rebaseline CRM Delta Pack v1.0

Document purpose
This document is a consolidated change-resolution and delta pack for the minimum authoritative re-baseline required to change the MVP camera ingest path from RTSP IP camera ingest to CSI/local-device ingest on the Jetson appliance.

Document status
Guiding change-control and merge artifact. Not authoritative by itself.

Primary use
Review, approve, and merge section-level changes into the authoritative source set:
- Consolidated Interface Control Document v1.1
- Adaptive Retail Golden Roadmap v2.1
- Pilot Validation Protocol v1.1

Why this document exists
The current authoritative ICD baseline formalizes ICD-1 as Camera (RTSP server) to DeepStream (RTSP client) and requires an rtsp_url field in camera source configuration. The roadmap requires service work to execute against the current ICD baseline and requires versioned contract updates and conformance testing for interface changes. The pilot protocol prohibits interface or schema drift during the pilot window. Therefore, changing the MVP camera path to CSI/local-device ingest requires a deliberate, versioned re-baseline rather than an informal implementation deviation.

Change summary
Decision:
The MVP single-camera pilot baseline is changed from RTSP IP camera ingest to CSI/local-device ingest on the Jetson appliance.

Reason:
The single-board Ethernet port is reserved for the local router or LAN path. Preserving an RTSP-over-Ethernet camera path would add avoidable network hardware, cost, support burden, and failure points for a one-camera appliance. A local CSI camera path is within the architecture envelope already contemplated by the SAD, while better fitting the locked MVP shape.

Change class
Major interface baseline update to ICD-1 and dependent execution/pilot references.

Affected authoritative documents
1. Consolidated Interface Control Document v1.0 -> v1.1
2. Adaptive Retail Golden Roadmap V2 -> v2.1
3. Pilot Validation Protocol v1.0 -> v1.1

Out of scope for this delta pack
- Rewriting unaffected sections
- Revising TRD thresholds unless later testing proves necessary
- Reworking downstream metadata contracts unless a CSI-specific timestamp or source-field clarification is required
- Rewriting the SAD, which already allows CSI/local-device ingest at the template level

Change ID
ICD-1-CSI-001

Section reference
Architecture context / Relevant components

Old language
"Relevant components (minimal context needed to implement these interfaces): - Camera: IP camera(s) streaming video to the edge network using RTSP. RTSP 2.0 is a standardized control protocol for media streaming sessions. - CV Pipeline (DeepStream on Jetson): DeepStream consumes the RTSP stream, performs inference/tracking, and emits metadata only."

New language
"Relevant components (minimal context needed to implement these interfaces): - Camera: a single local camera connected directly to the Jetson through the approved MVP hardware interface, currently CSI and exposed to the ingest pipeline as a local video device. - CV Pipeline (DeepStream on Jetson): DeepStream consumes the approved local camera device stream, performs inference/tracking, and emits metadata only."

Reason for change
The active MVP appliance is being re-baselined from RTSP IP camera ingest to direct local-camera ingest for the single-camera pilot. This removes unnecessary network hardware dependence while remaining inside the architecture envelope already contemplated by the SAD.

Downstream implications
- Removes implied dependence on IP camera deployment for the MVP baseline
- Aligns camera prose with the new ICD-1 contract
- Does not change downstream metadata-only privacy posture

Required conformance updates
- Update implementation assumptions in input-cv launcher/config loader
- Add local-device ingest validation tests

Change ID
ICD-1-CSI-002

Section reference
ICD camera to CV pipeline / Interface name: ICD-1 Camera -> CV Pipeline / Primary purpose / Actors / Transport protocol / Addressing or endpoint form

Old language
"Interface name: ICD-1 Camera -> CV Pipeline
Primary purpose: Deliver a continuous video stream from an IP camera to the edge CV pipeline for inference.
Actors: - Producer: Camera (RTSP server) - Consumer: DeepStream pipeline (RTSP client)
Transport protocol: - RTSP 2.0 for session control (default).
Addressing / endpoint form: - RTSP URL (canonical form; exact path is camera-specific): - rtsp://<host>:<port>/<path> - rtsp://<username>:<password>@<host>:<port>/<path>"

New language
"Interface name: ICD-1 Camera -> CV Pipeline
Primary purpose: Deliver a continuous video stream from the approved local MVP camera device to the edge CV pipeline for inference.
Actors: - Producer: Local camera device presented by the Jetson host OS - Consumer: DeepStream pipeline (local-device client)
Transport protocol: - Local device ingest through the approved host camera interface for the MVP baseline, currently V4L2-compatible local device semantics for CSI-connected capture.
Addressing / endpoint form: - Local device path and capture parameters, for example: /dev/video0"

Reason for change
The prior RTSP/IP-camera framing is no longer the MVP baseline. The new baseline uses a camera attached directly to the board and consumed as a local device.

Downstream implications
- Removes RTSP-only assumptions from input-cv
- Shifts deployment complexity from network addressing to local device enumeration and capture configuration

Required conformance updates
- New ingest configuration parser and validation
- New bring-up checklist for local device path, format, fps, permissions, and startup ordering

Change ID
ICD-1-CSI-006

Section reference
ICD-1 / Error and failure modes / Retry-backoff-timeouts

Old language
"Authentication failure (camera rejects credentials). RTSP session negotiation failure (DESCRIBE/SETUP/PLAY failures). Stream stalls / read timeout. Jitter, packet loss, decoder errors. Time drift ... On connect failure: retry with exponential backoff ... On read timeout: treat as transient failure; teardown session and reconnect."

New language
"Failure modes for the MVP local-camera baseline include: local device missing at startup; device node permission failure; unsupported pixel format or resolution; device read stall; hot-unplug or camera disconnect; intermittent frame timeout; and timestamp inconsistency if the device clock or capture timestamps are unstable.

On startup failure: retry open with exponential backoff within the configured reopen window.
On read timeout or stall: treat as transient failure; close and reopen the local device.
If failures exceed a site-defined threshold: degrade gracefully by emitting camera-offline health status in ICD-2 and preserving playback fallback behavior."

Reason for change
Failure semantics must reflect a local-device camera path rather than an RTSP session.

Downstream implications
- Supervisor and input-cv health reporting should map local-device failures cleanly into camera-offline state

Required conformance updates
- Missing-device, stall, and reopen tests
- Camera disconnect recovery drill
